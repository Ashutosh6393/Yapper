Yapper Architecture

A collaborative, real-time, rich-text note-taking app. Three apps in a Bun/Turborepo monorepo, sharing typed packages, backed by Postgres + Redis, with conflict-free real-time editing over Yjs and stateless WebSocket auth.

Bun + Turborepo Next.js · TipTap · Yjs Hocuspocus Better Auth + JWT/JWKS Drizzle · PostgreSQL Redis fanout

Overview

Yapper splits cleanly into three deployable apps plus a set of shared packages. The web app is the editor and dashboard, the api app owns identity, notes metadata and sharing, and the socket app is the realtime document backend. The two server apps never duplicate domain logic — permission derivation, the editor schema, the DB client, and auth verification all live in shared packages.

web Next.js · TS · Tailwind

Hosts the TipTap editor bound to a Yjs document, the @hocuspocus/provider WebSocket client, and the Better Auth React client. Renders dashboard + editor pages.

api Bun · Express

Mounts the Better Auth handler at /api/auth/* (Google + GitHub OAuth). Owns notes & sharing REST endpoints, note metadata + collaborator records. On "make private", publishes a revoke event to Redis.

socket Bun · Hocuspocus

Yjs WebSocket backend. Hooks: onAuthenticate (verify JWT + check permission), onLoadDocument/onStoreDocument (persist Yjs state), @hocuspocus/extension-redis for cross-instance fanout, listens for revoke events to close connections.

shared packages db · auth · editor · permissions

db (Drizzle schema+client), auth (Better Auth config + JWT/JWKS verify), editor (TipTap schema + doc→title/preview), permissions (effective-permission derivation + Redis cache), typescript-config.

Design principle: permissions are derived, never stored per person, and the same derivation code runs in both api (REST gating) and socket (onAuthenticate). Presence identity is server-authoritative. Persistence is a single debounced full-state blob.

01 System / Component Diagram

The browser talks to two backends: plain HTTP/REST to api for auth and notes metadata, and a persistent WebSocket to socket for the live document. Both server apps read/write Postgres through the shared db package. Multiple socket instances stay in sync through Redis, which carries Yjs updates, awareness, revoke broadcasts, and the permissions cache.

02 Monorepo Layout & Dependencies

03 Auth + JWT Sequence

Login is a classic Better Auth OAuth round-trip handled entirely by api, ending in a cookie session. For the WebSocket leg, the client requests a short-lived JWT from Better Auth's JWT plugin and hands it to Hocuspocus as the connection token. The socket app verifies it statelessly against the JWKS endpoint — no session lookup, no DB hop — then derives permission inside onAuthenticate.

The userId claim flows out of the verified JWT directly into onAuthenticate and becomes the server-authoritative identity used for presence stamping.

04 Real-Time Collaboration Sequence

Two editors on the same note connect to (possibly different) socket instances. Local edits become Yjs updates, broadcast to peers on the same instance and fanned out across instances via Redis. Awareness (cursors/selection) flows the same way but is never persisted. The document state is written to Postgres on a debounced onStoreDocument (~2s), at which point title + preview are re-derived and denormalized onto the note row.

Viewers are presence-only — they appear in awareness but do not broadcast caret/selection, and the server drops any inbound doc updates from a read-only connection.

05 Database Schema (ER)

Better Auth owns the identity tables (user, session, account, verification). Yapper adds note (metadata + access enum + share token), note_doc (the single full-state Yjs blob), and note_collaborator (membership with active/revoked status). Effective permission is computed from these — there is no per-person permission column.

06 Make-Private / Revoke Flow

When an owner makes a note private, api updates state and busts caches, then publishes a revoke:{noteId} event to Redis. Every socket instance holding that note's room closes the affected non-owner connections. Each client's provider sees the close / auth-failure and routes the user out with a clear message. The owner's own connection is untouched.

Owner toggles private → fanout → forced disconnect

sequenceDiagram
    autonumber
    participant O as Owner (web)
    participant A as api
    participant DB as PostgreSQL
    participant R as Redis
    participant S as socket instances
    participant C as Collaborator (web)

    O->>A: PATCH /notes/:id { access: private }
    A->>DB: set access=private
    A->>DB: rotate/invalidate share_token
    A->>DB: mark note_collaborator rows revoked
    A->>R: bust permissions cache (noteId)
    A->>R: publish revoke:{noteId}
    A-->>O: 200 OK (owner stays connected)

    R-->>S: revoke:{noteId}
    S->>S: close all non-owner connections in room
    S-->>C: connection closed / auth-failed
    C->>C: provider onClose → "note made private by owner"
    C->>C: route user out of editor

Cache-bust before publish guarantees that any reconnection attempt re-derives permission from fresh state and resolves to none.

07 Permission Derivation

Effective permission is a pure function of (userId, noteId) against the note's owner, access enum, and collaborator status. This single derivation lives in packages/permissions and is consumed identically by api (REST gating) and socket (onAuthenticate), with results cached in Redis and busted on any access change.

    class Edit,View,None res;

edit

Owner always; or access='edit' + active collaborator. Full read/write Hocuspocus connection, broadcasts caret + selection.

view

access='view' + active collaborator. Server resolves a readOnly connection: inbound doc updates dropped, outbound + presence still streamed. Presence-only.

none

Private note, or non-collaborator. onAuthenticate rejects the connection; REST endpoints return forbidden.

shared logic

One implementation in packages/permissions → no drift between REST gating and WS auth. Role changes force client reconnection so the new permission takes effect.

Deployment

Now — local-dev-first. Docker Compose runs Postgres + Redis; all three apps run via turbo dev. The Redis fanout works the same with a single instance, so the realtime path is exercised locally.

  • Phase 1 (current): Docker Compose (Postgres + Redis) + turbo dev for web/api/socket.
  • Phase 2: single-platform containers (Railway / Render) — one container per app, managed Postgres + Redis.
  • Phase 3: split topology — web on Vercel, api + socket as containers, scaled horizontally. The Redis fanout is what makes multiple socket instances coherent.
Why Redis from day one: even though local dev needs only one socket instance, all cross-instance coordination (Yjs updates, awareness, revoke broadcast, permission cache) routes through Redis so horizontal scaling in Phase 3 requires no architectural change.
